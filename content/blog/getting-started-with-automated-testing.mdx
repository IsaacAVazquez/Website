# Getting Started with Automated Testing

Automated testing is one of those topics where the advice is almost universally correct and almost universally unhelpful at the same time. Everyone knows you should write tests. The harder question is where to start, what to test first, and how to build something you will actually maintain.

The pattern I keep coming back to is the testing pyramid. Not because it is a law of nature, but because it reflects something true about tradeoffs. Unit tests are cheap to write and fast to run, so you should have a lot of them. End-to-end tests are expensive and slow, so you should use them sparingly on the workflows that actually matter. Integration tests sit in the middle and cover the boundaries between components, which is where a surprising number of real bugs live.

In practice, I aim for roughly 70% unit tests, 20% integration tests, and 10% end-to-end. The specific numbers matter less than the underlying logic: most of your feedback should come from fast, isolated tests, with E2E coverage reserved for critical paths.

## Tools

For JavaScript and TypeScript projects, I use Jest for unit testing, Testing Library for component testing, and Playwright for end-to-end. Playwright has become my default over Cypress for new projects because of its cross-browser support and better handling of async behavior. For API testing within the Node ecosystem, Supertest is the straightforward choice.

## What Makes Tests Actually Maintainable

The most common failure mode I see is tests that are tightly coupled to implementation details. When you refactor code, those tests break even though the behavior has not changed, and then people start treating the test suite as noise rather than signal.

The fix is to test behavior, not internals. A test for user authentication should verify that a valid user can log in and receive a token. It should not care which function handles the password hash.

```typescript
describe('User Authentication', () => {
  test('allows valid user to login', async () => {
    const user = await createTestUser();
    const response = await login(user.email, user.password);

    expect(response.status).toBe(200);
    expect(response.body.token).toBeDefined();
  });
});
```

Clean test data is the other half of this. Tests that share state or depend on a specific database condition end up brittle and hard to debug in isolation. I clean and reseed before each test:

```typescript
beforeEach(async () => {
  await cleanDatabase();
  await seedTestData();
});
```

On assertions, specificity matters. A vague assertion like `toBeTruthy()` will pass even when something is subtly wrong. This is better:

```typescript
expect(response.body).toEqual({
  id: expect.any(Number),
  email: 'test@example.com',
  status: 'active',
  createdAt: expect.any(String)
});
```

## Flaky Tests Are the Biggest Drain on a Test Suite

A test that sometimes passes and sometimes fails is often worse than no test at all. It conditions people to ignore failures. Most flakiness traces back to a few things: missing waits in async flows, test data bleeding between runs, or dependencies on external services that are not reliably available. The answer is to find the root cause rather than adding retries. Retries mask the problem and make the suite slower.

## Beyond Functional Testing

Visual regression testing is worth adding for any UI that is stable enough to capture. Playwright makes this relatively low-friction:

```typescript
test('homepage visual regression', async ({ page }) => {
  await page.goto('/');
  await expect(page).toHaveScreenshot('homepage.png');
});
```

For APIs or pages with performance requirements, a simple timing assertion catches regressions before they become user-facing problems:

```typescript
test('API response time under 500ms', async () => {
  const start = Date.now();
  await fetch('/api/users');
  expect(Date.now() - start).toBeLessThan(500);
});
```

## The Real Goal Is Confidence

100% code coverage is not the goal. The goal is a test suite you trust, that runs fast enough to stay out of your way, and that catches the things that actually matter. Start with your most critical user paths, write tests that focus on behavior, and treat flakiness as a bug to fix rather than a cost of doing business.
