# QA Engineering: Building Quality into Software Development

QA engineering is one of those roles that looks simple from the outside and turns out to be surprisingly deep once you're in it. The job description says "find bugs," but the actual work is about building systems and processes that make bugs visible earlier, reduce the cost of finding them, and give development teams enough confidence to ship at speed. Those are different problems.

I've built testing infrastructure for high-traffic applications, from civic tech platforms serving tens of millions of voters in Austin to more recent work at Silicon Valley scale. Alongside that, I've been doing my MBA at UC Berkeley, which has changed how I think about QA as a business function rather than just a technical one. Both lenses matter.

## What QA Actually Means

Quality assurance is a systematic approach to ensuring software meets requirements and functions as intended. But the more useful framing is that QA is about risk management. When you decide what to test, how thoroughly to test it, and when it's good enough to ship, you're making risk decisions. The question is whether those decisions are explicit and defensible or implicit and arbitrary.

The field has shifted significantly over the last decade. In waterfall-era development, testing happened at the end of a long cycle, QA teams worked in isolation, and feedback loops were measured in weeks. In modern agile and DevOps environments, testing is integrated throughout development, QA engineers are embedded in teams, and feedback loops need to be measured in minutes. The skills required are different, and the relationship to the codebase is different.

What's stayed constant is the core function: QA engineers are the people asking "but what happens when..." on behalf of users who will never have the same mental model of the system as the people who built it.

## The Testing Pyramid

The testing pyramid is the most useful framework I know for thinking about test suite structure, and it's held up well despite being old.

```
    /\     E2E Tests (10%)
   /  \    - Slow, brittle, expensive
  /____\   - Full user workflows
 /      \
/________\ Integration Tests (20%)
          - API contracts, service interactions
          - Moderate speed and cost

__________ Unit Tests (70%)
          - Fast, reliable, cheap
          - Individual function testing
```

The rationale is economic. Unit tests are cheap to write, fast to run, and precise about what broke. End-to-end tests are expensive, slow, and when they fail they often don't tell you why. You want the expensive tests to be the ones covering things unit tests structurally can't cover, such as full user workflows, cross-browser behavior, and integration points with external services.

I've seen teams invert this accidentally. They write lots of E2E tests because they feel thorough and concrete, then spend half their CI time maintaining flaky selectors. The result is a test suite that's slow to run, slow to fix, and loses credibility over time because it fails for reasons unrelated to real bugs.

## Test Strategy

Before writing a single test, it's worth being explicit about what you're trying to protect. I think in terms of risk: what breaks, how often, and what it costs when it does. That drives priority.

High-priority testing areas are generally critical business functions (payment processing, authentication), recently changed code, integration points with external systems, and anything security-sensitive. Not because those are always the most interesting to test, but because failures there are the most expensive.

The testing types worth knowing well are unit testing for individual functions in isolation, integration testing for component interactions and API contracts, system testing for complete workflows, and acceptance testing for business requirements. On the non-functional side, performance testing (load, stress, scalability), security testing, and accessibility testing all tend to get treated as optional until they're not.

Test-Driven Development is worth understanding even if you don't practice strict red-green-refactor. The discipline of writing a test before writing code forces you to think about the interface before the implementation, which tends to produce more testable code.

```javascript
// 1. Write failing test
describe('calculateTotal', () => {
  test('should add tax to subtotal', () => {
    expect(calculateTotal(100, 0.08)).toBe(108);
  });
});

// 2. Write minimal code to pass
function calculateTotal(subtotal, taxRate) {
  return subtotal + (subtotal * taxRate);
}

// 3. Refactor and improve
function calculateTotal(subtotal, taxRate) {
  if (subtotal < 0 || taxRate < 0) {
    throw new Error('Values must be non-negative');
  }
  return Number((subtotal * (1 + taxRate)).toFixed(2));
}
```

BDD (Behavior-Driven Development) is useful when you want tests that double as documentation and can be read by non-engineers:

```gherkin
Feature: Shopping Cart Checkout
  As a customer
  I want to calculate my total with tax
  So that I know how much to pay

Scenario: Calculate total with tax
  Given I have items worth $100 in my cart
  And the tax rate is 8%
  When I proceed to checkout
  Then my total should be $108.00
```

## Automation

A question I get asked a lot is when to automate. My honest answer is that you automate anything you need to run repeatedly and reliably. Regression testing, critical path testing, data-driven scenarios, and API contract testing are all good candidates. One-time checks, exploratory testing, usability testing, and anything with a UI that changes frequently are often not worth automating, at least not aggressively.

The JavaScript/TypeScript ecosystem has solid tooling. Jest for unit and integration tests, React Testing Library for component tests, Playwright for E2E. Here's what each looks like in practice:

```javascript
// Jest unit test
import { formatCurrency } from '../utils/currency';

describe('formatCurrency', () => {
  test('formats positive numbers correctly', () => {
    expect(formatCurrency(1234.56)).toBe('$1,234.56');
  });

  test('handles zero correctly', () => {
    expect(formatCurrency(0)).toBe('$0.00');
  });

  test('handles negative numbers', () => {
    expect(formatCurrency(-100)).toBe('-$100.00');
  });
});
```

```javascript
// React Testing Library component test
import { render, screen, fireEvent, waitFor } from '@testing-library/react';
import { UserProfile } from '../components/UserProfile';

describe('UserProfile', () => {
  test('displays user information correctly', () => {
    const user = { name: 'John Doe', email: 'john@example.com' };
    render(<UserProfile user={user} />);

    expect(screen.getByText('John Doe')).toBeInTheDocument();
    expect(screen.getByText('john@example.com')).toBeInTheDocument();
  });

  test('handles edit mode toggle', async () => {
    const user = { name: 'John Doe', email: 'john@example.com' };
    render(<UserProfile user={user} />);

    fireEvent.click(screen.getByRole('button', { name: /edit/i }));

    await waitFor(() => {
      expect(screen.getByRole('textbox', { name: /name/i })).toBeInTheDocument();
    });
  });
});
```

```javascript
// Playwright E2E test
import { test, expect } from '@playwright/test';

test.describe('User Authentication', () => {
  test('should login with valid credentials', async ({ page }) => {
    await page.goto('/login');

    await page.fill('[data-testid="email-input"]', 'test@example.com');
    await page.fill('[data-testid="password-input"]', 'password123');
    await page.click('[data-testid="login-button"]');

    await expect(page).toHaveURL('/dashboard');
    await expect(page.locator('[data-testid="user-menu"]')).toBeVisible();
  });

  test('should show error for invalid credentials', async ({ page }) => {
    await page.goto('/login');

    await page.fill('[data-testid="email-input"]', 'invalid@example.com');
    await page.fill('[data-testid="password-input"]', 'wrongpassword');
    await page.click('[data-testid="login-button"]');

    await expect(page.locator('[data-testid="error-message"]'))
      .toContainText('Invalid credentials');
  });
});
```

The Page Object pattern is worth adopting for E2E tests once your suite has more than a few tests. It keeps locators centralized so when the UI changes you update one file instead of twenty:

```javascript
// pages/LoginPage.js
export class LoginPage {
  constructor(page) {
    this.page = page;
    this.emailInput = '[data-testid="email-input"]';
    this.passwordInput = '[data-testid="password-input"]';
    this.loginButton = '[data-testid="login-button"]';
    this.errorMessage = '[data-testid="error-message"]';
  }

  async goto() {
    await this.page.goto('/login');
  }

  async login(email, password) {
    await this.page.fill(this.emailInput, email);
    await this.page.fill(this.passwordInput, password);
    await this.page.click(this.loginButton);
  }

  async getErrorMessage() {
    return await this.page.textContent(this.errorMessage);
  }
}
```

For API testing, Supertest is the straightforward choice for Node.js:

```javascript
import request from 'supertest';
import app from '../app';

describe('User API', () => {
  test('POST /users should create new user', async () => {
    const userData = {
      name: 'John Doe',
      email: 'john@example.com',
      password: 'securePassword123'
    };

    const response = await request(app)
      .post('/api/users')
      .send(userData)
      .expect(201);

    expect(response.body).toMatchObject({
      id: expect.any(Number),
      name: userData.name,
      email: userData.email
    });
    expect(response.body.password).toBeUndefined();
  });
});
```

## CI/CD Integration

Tests that don't run automatically don't get run. A CI pipeline that runs unit tests, integration tests, and E2E tests on every pull request is the baseline:

```yaml
name: CI/CD Pipeline

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2

      - name: Setup Node.js
        uses: actions/setup-node@v2
        with:
          node-version: '18'

      - name: Install dependencies
        run: npm ci

      - name: Run linting
        run: npm run lint

      - name: Run unit tests
        run: npm run test:unit

      - name: Run integration tests
        run: npm run test:integration

      - name: Run E2E tests
        run: npm run test:e2e

      - name: Generate test coverage
        run: npm run coverage
```

## Advanced Testing: Performance, Security, Accessibility

Performance testing gets skipped until it becomes a crisis. Load testing with a tool like Artillery lets you simulate realistic traffic before it's a production problem:

```yaml
config:
  target: 'https://api.example.com'
  phases:
    - duration: 60
      arrivalRate: 10
    - duration: 120
      arrivalRate: 50
    - duration: 60
      arrivalRate: 10
```

Security testing for QA engineers usually means covering the obvious attack surface in automated tests (SQL injection, XSS, authentication boundaries):

```javascript
describe('Security Tests', () => {
  test('should prevent SQL injection in user search', async () => {
    const maliciousInput = "'; DROP TABLE users; --";

    const response = await request(app)
      .get(`/api/users/search?q=${encodeURIComponent(maliciousInput)}`)
      .expect(400);

    expect(response.body.error).toContain('Invalid search query');
  });

  test('should sanitize XSS attempts', async () => {
    const xssPayload = '<script>alert("xss")</script>';

    const response = await request(app)
      .post('/api/comments')
      .send({ text: xssPayload })
      .expect(201);

    expect(response.body.text).not.toContain('<script>');
    expect(response.body.text).toContain('&lt;script&gt;');
  });
});
```

Accessibility testing is underinvested across the industry. Automated tools catch maybe 30% of issues, but they catch them at low cost:

```javascript
import { axe, toHaveNoViolations } from 'jest-axe';
import { render } from '@testing-library/react';

expect.extend(toHaveNoViolations);

test('should not have accessibility violations', async () => {
  const { container } = render(<MyComponent />);
  const results = await axe(container);
  expect(results).toHaveNoViolations();
});
```

## Team Integration

QA doesn't work well in isolation. The most effective pattern I've seen is QA engineers embedded in development teams, participating in sprint planning to review user stories for testability, and involved in code review to catch things that will be hard to test after the fact. The code review angle is underrated:

```javascript
// Original code
function processPayment(amount) {
  return amount * 1.08; // tax calculation
}

// After QA review, input validation added, tax rate made configurable,
// floating-point precision handled
function processPayment(amount, taxRate = 0.08) {
  if (typeof amount !== 'number' || amount < 0) {
    throw new Error('Amount must be a positive number');
  }

  if (typeof taxRate !== 'number' || taxRate < 0) {
    throw new Error('Tax rate must be a positive number');
  }

  return Number((amount * (1 + taxRate)).toFixed(2));
}
```

## Quality Metrics

Coverage is the metric people default to, but it's a proxy for what you actually care about, which is bug detection effectiveness. A codebase can have 90% line coverage and still miss the scenarios that matter. I track defect density, escaped defect rate (bugs found in production vs. pre-production), and mean time to detection. These measure outcomes, not activity.

```javascript
const qualityMetrics = {
  testCoverage: {
    lines: 85.4,
    branches: 78.2,
    functions: 92.1,
    statements: 87.6
  },
  defectMetrics: {
    totalDefects: 23,
    criticalDefects: 2,
    defectDensity: 0.12,
    escapedDefects: 1
  },
  testExecution: {
    totalTests: 1247,
    passedTests: 1243,
    failedTests: 4,
    skippedTests: 0,
    executionTime: '4m 32s'
  }
};
```

## Austin vs. Bay Area

I've worked in both markets and they have genuinely different cultures around quality. In Austin's civic tech sector, the framing is impact-focused. Quality decisions have direct consequences for civic participation, so there's a strong orientation toward sustainable, maintainable practices. In the Bay Area, the dominant concern is scale and velocity. Quality processes have to enable speed, not slow it down, and everything is expected to be instrumented and data-driven.

I think the best QA engineers draw on both. Austin's emphasis on long-term maintainability keeps you from building test suites that collapse under their own weight. The Bay Area's data-driven culture keeps you honest about whether your testing is actually finding bugs.

## What's Coming

AI-assisted testing, shift-left practices, and cloud-native testing approaches are real trends, not just buzzwords. The practical implication for QA engineers is that the job will keep moving away from execution and toward strategy. The engineers who understand what AI testing tools can and can't do, and who can make quality arguments in business language, not just technical language, are going to be the ones setting the direction for their teams.

That's a harder job to do well, and also a more interesting one.
