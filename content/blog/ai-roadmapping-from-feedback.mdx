# Roadmapping with AI: Turning Raw Feedback into Quarterly Bets

The hardest part of roadmap planning is not prioritization. Most frameworks for weighing impact against effort are not that complicated. The hard part is the step before that: taking the raw signal you have collected from user interviews, support tickets, sales feedback, and internal requests and turning it into a coherent set of options that are actually comparable.

Without that synthesis step being done well, the prioritization process is doing more work than it should. You end up arguing about themes rather than tradeoffs because the underlying feedback has not been processed into a form where the tradeoffs are clear. AI has changed this step substantially for me, and the way it changed it is specific enough to describe.

## What the raw input actually looks like

Before I can synthesize feedback into roadmap inputs, I have to have the feedback in a usable form. This sounds obvious, and it took me longer than it should have to get right.

In practice, my raw input for a quarterly planning cycle comes from several sources with different formats and different levels of reliability. User interview transcripts are the highest quality signal, with specific quotes and context about what users were doing and why. Support ticket themes are aggregated and noisy, but they represent actual pain points at scale. Sales win/loss notes reflect what has been told to me about what matters to buyers, filtered through another person's interpretation. Internal requests come with political weight that does not always correlate with user value.

The mistake I made early on was treating all of this as equivalent. It is not. A single user saying something in an interview is a data point. Fifteen support tickets describing the same workflow breakdown is a pattern. A sales team saying customers keep asking about integration X might be genuine or might be a negotiating tactic that never shows up after the deal is signed. The synthesis step has to preserve these distinctions, not flatten them.

## Setting up the synthesis

The workflow I use now starts with a structure document before I involve AI at all. I spend an hour or two organizing my raw inputs by type and by what I already believe about their relative reliability. I write brief notes on each major input source: how it was collected, who was involved, what biases might be in it, and what questions it leaves open.

That hour is not optional. When I have tried to jump straight to AI synthesis without doing it, the outputs are plausible but shallow. The model surfaces the themes that are easiest to find rather than the ones that matter most. The pre-work forces me to engage with the source material myself before I hand it off for pattern recognition.

## Clustering problems, not solutions

The first AI pass is clustering. I give Claude my organized input and ask it to identify the distinct problems users are describing, with direct quotes as evidence, grouped by similarity and tagged with source type. I specifically ask it not to infer solutions, just to describe the problems as users experienced them.

The result is usually a set of 8 to 15 problem clusters, with some that are clearly the same problem described from different angles and some that are related but distinct. I review this output manually and collapse the ones that are clearly overlapping, separate the ones that got merged when they should not have, and cut the ones that represent edge cases that do not warrant roadmap attention.

This takes about 30 minutes and produces what I think of as the cleaned feedback inventory: the set of problems worth taking into the prioritization discussion.

I want to be specific about why asking for problems rather than solutions matters. If I let the AI suggest solutions, it will surface the obvious ones, and the obvious solutions will anchor the subsequent discussion. By clustering problems first and generating solution directions separately, I preserve the ability to consider non-obvious approaches to real problems.

## Estimating relative scale

The next step is harder, and AI is less reliable here, but still useful.

I want some sense of how many users are affected by each problem and how significant the impact is when they encounter it. This is inherently an estimation exercise because I almost never have clean data for it. What I have is a mix of high-quality interviews that give depth but not breadth, support ticket volumes that give relative frequency, and some user research on task frequency and criticality.

I describe each problem cluster to Claude and give it the evidence I have for scale and impact. Then I ask it to flag which estimates feel underconfident and what additional information would most change the picture. This produces a set of questions I can answer quickly, usually by pulling a number from analytics or by scheduling a targeted user call. The model is useful here not because it can estimate the number, but because it identifies where the number is missing and what it would take to get a better one.

## Writing option descriptions

By the time I reach the actual roadmap discussion, I want to have already turned feedback themes into option descriptions. An option is not a feature request. It is a problem statement, a proposed solution direction, a rough sense of who it affects, and a hypothesis about why it matters strategically.

AI is good at the option description step. I give it a problem cluster, the evidence for it, and a rough solution direction I have sketched out myself, and ask it to write a clear option description that would make sense to a stakeholder who was not in the user research. I then edit that description to add anything that requires context it does not have.

The goal is not to have AI make the strategic argument for the option. That argument is mine, and I need to be able to make it in my own words or it will not survive the discussion. What AI does is save the drafting time so I can get more options to the description stage before the meeting.

## What AI cannot do in this process

The judgment calls that roadmapping actually requires are not in the synthesis steps. They are in interpreting conflicting evidence, deciding which strategic bets align with where the company is going, and making the call about which problem to solve when you cannot solve all of them.

I have seen PMs try to outsource these calls. They give a language model the list of options, a set of criteria, and ask it to recommend the priority order. The recommendation is usually defensible and usually wrong, because the model does not understand the history behind why certain bets are off the table or how the upcoming competitive environment changes what matters.

The synthesis work that AI accelerates is genuinely valuable. It compresses the time between collecting feedback and having a structured set of options from several days to a few hours. But the time savings in synthesis only matters if you spend the time freed on the judgment work that synthesis was clearing the path for. The roadmap does not get better because the clustering was faster. It gets better because the faster clustering gave you more time to think carefully about what the options mean.
