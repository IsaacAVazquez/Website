# Building a GitHub Trending Dashboard That Is Not Just a Leaderboard

If you want to see the product before reading the breakdown, open [GitHub Trending Pulse](/github-trending-pulse).

Everyone who writes code keeps a loose mental list of what is happening in open source, and almost nobody has a good way to maintain it. GitHub itself has a trending page, but it is a single ranked list that resets every day, and a list is a strange way to understand momentum. What I actually want to know is not which repository has the most stars, because the answer to that has been the same set of giant projects for years. I want to know what is moving, in which language, around which topic, and whether it is accelerating or already cooling off. So I built GitHub Trending Pulse to treat trending as a shape rather than a leaderboard.

## Discovery on GitHub is noisier than it should be

The frustrating thing about finding interesting projects on GitHub is that the relevant signals all live in different places. The genuinely popular projects sit at the top of any all-time star search and rarely move, the freshly active repositories are buried under that weight, and topic-specific momentum, the stuff happening right now in one niche, requires a search query you have to construct by hand. None of those three views talk to each other, so building an accurate picture of what is happening means running several searches and holding the results in your head. That is a lot of effort to answer a question I ask almost every week.

So the dashboard does the organizing instead of the reader. It pulls from the public GitHub Search API and arranges the results into fourteen segments split across languages and topics, so the page is not one list but a set of focused windows into different corners of the ecosystem. Looking at what is active in a specific language is a different question from looking at what is active around a specific topic, and the segmented layout lets you ask either one without writing a query. The value is in the curation of those windows, not in reprinting a single ranking that anyone could already find.

## Why it shows movement, not totals

This is the decision that makes the product different from the official trending page, and it is the one I care most about. A single daily list can only ever show you a state, which is who is on top today. It cannot show you change, because it has no memory of yesterday. GitHub Trending Pulse carries weekly star deltas forward across days, so each repository shows how its stars have moved over the trailing week, not just its all-time count. That difference sounds small and is actually the whole thesis. An all-time star count tells you what was popular at some point in history. A weekly delta tells you what is gaining attention right now, and a delta tracked across several days tells you whether that attention is accelerating or fading.

Acceleration is the signal that genuinely useful discovery is built on, and it is exactly the thing a snapshot with no history cannot give you. A project that added a few thousand stars this week is a different story from one that has been quietly adding the same number every week for a year, and you can only tell those apart if the data remembers. By carrying the deltas forward, the dashboard turns a flat ranking into something closer to a velocity reading, which is what I actually wanted from a trending tool in the first place.

## The snapshot that makes the memory possible

The dashboard reads from a snapshot committed to the repository as a plain TypeScript file, generated by a build script that queries the GitHub Search API, shapes the fourteen segments, and writes the file. A scheduled job refreshes it daily, and the running site never calls GitHub at request time. The snapshot pattern is not just about reliability here, although it gives me that too, with instant page loads and no risk of hitting an API rate limit while someone is reading. It is what makes the weekly movement possible at all. Because each refresh commits a dated snapshot, the data has a history, and that history is precisely what lets the dashboard show deltas instead of a stateless ranking. A live API call at request time would only ever know about right now. A committed daily snapshot knows about the trend.

The refresh follows the same conservative rule as the rest of the family, so a failed or empty fetch keeps the previous snapshot rather than wiping good data, which means a bad GitHub response cannot blank the board. For a developer who pops in once a week to see what is new, the worst outcome would be an empty page, so the pipeline always prefers the last good snapshot over nothing.

## What I took away from it

The thing this project clarified for me is that a leaderboard and a trend are different products that happen to look similar. A leaderboard answers "what is biggest," which is a question with a stable and slightly boring answer. A trend answers "what is moving," which is the question developers actually care about, and answering it well requires memory that a single live list does not have. Once I framed the build around movement instead of totals, the snapshot stopped being a convenience and became the core mechanism, because owning the data over time is the only way to show change over time. That reframing is the whole reason the dashboard is worth opening instead of just visiting GitHub's own trending page.

You can explore the full thing at [GitHub Trending Pulse](/github-trending-pulse). The quickest way to understand it is to scan a couple of language segments, then a topic segment, and pay attention to the weekly movement next to each repository rather than its raw star count, because that movement is the entire point of the project.
